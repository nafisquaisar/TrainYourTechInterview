📘 DBMS Normalization Notes

1. What is Dependency?

  • Functional Dependency: A relationship where one attribute uniquely determines another.

    • Example: order_id → customer_name (order_id determines the customer).

  • Partial Dependency: When a non-key attribute depends only on part of a composite key.

  • Transitive Dependency: When a non-key attribute depends on another non-key attribute instead of directly on the key.

  • Key Attributes→ Columns that are part of the Primary Key.
    👉 In your table: order_id, book_isbn

  • Composite Key → When the Primary Key is made of 2 or more columns.
    👉 In your table: (order_id, book_isbn)

  • Non-Key Attributes → All other columns that are not part of the Primary Key.
    👉 In your table: customer_name, customer_email, customer_address, book_title, book_author, book_price, order_date, quantity, total_price

2. What is Normalization?

Normalization is the process of organizing data in a database to reduce redundancy and improve data integrity.

Benefits:

  • Avoids data duplication.
  • Ensures consistency.
  • Makes maintenance easier.

3. Types of Normal Forms

  1. 1NF (First Normal Form)

    • No repeating groups.
    • Atomic values only.

  2. 2NF (Second Normal Form)

    • Must be in 1NF.
    • No partial dependency (all non-key attributes depend on the whole primary key).

  3. 3NF (Third Normal Form)

    • Must be in 2NF.
    • No transitive dependency (non-key attributes must depend only on the primary key).

  4. BCNF (Boyce-Codd Normal Form)

    • Stronger version of 3NF.
    • Every determinant must be a super key.

6. Conversion to 3NF

The 2NF design still has transitive dependencies in the orders2NF table.

❌ Why not 3NF? (Transitive Dependencies)

  • In orders2NF table:

    • customer_email and customer_address depend on customer_name.
    • customer_name itself depends on order_id.

  • This creates a chain:
    order_id → customer_name → customer_email, customer_address

Example Proof

  • If John Smith changes his email, we’d need to update multiple rows.
  • The same customer always has the same email and address across orders.

7. 3NF Normalized Structure 

CREATE TABLE customers_3nf (
    customer_id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(100),
    address VARCHAR(255)
);

CREATE TABLE orders_3nf (
    order_id INT PRIMARY KEY,
    customer_id INT,
    order_date DATE,
    FOREIGN KEY (customer_id) REFERENCES customers_3nf(customer_id)
);

CREATE TABLE books_3nf (
    isbn VARCHAR(20) PRIMARY KEY,
    title VARCHAR(200),
    author VARCHAR(100),
    price DECIMAL(10, 2)
);

CREATE TABLE order_items_3nf (
    order_id INT,
    book_isbn VARCHAR(20),
    quantity INT,
    total_price DECIMAL(10, 2),
    PRIMARY KEY (order_id, book_isbn),
    FOREIGN KEY (order_id) REFERENCES orders_3nf(order_id),
    FOREIGN KEY (book_isbn) REFERENCES books_3nf(isbn)
);

9. ✅ 3NF Verification

  • No partial dependencies (already handled in 2NF).

  • No transitive dependencies:

    • customers_3nf: All attributes depend directly on customer_id.
    • orders_3nf: All attributes depend directly on order_id.
    • books_3nf: All attributes depend directly on isbn.
    • order_items_3nf: All attributes depend on the composite key (order_id, book_isbn).

✅ Now the schema is in 3NF.

1. 🔑 What is BCNF?

  • BCNF (Boyce-Codd Normal Form) is a stronger version of 3NF.

  • A table is in BCNF if:

    1. It is in 3NF.
    2. For every functional dependency (X → Y), the determinant X must be a superkey.
    • (X → Y) -> X functionally determines Y

⚠️ In some cases, a table may be in 3NF but not in BCNF, if a non-prime attribute (not part of a candidate key) is dependent on part of a key.

2. ⚡ Example of 3NF but not BCNF

Consider a student_courses table:

student_id course_id instructor
1 DBMS Prof. A
2 DBMS Prof. A
3 Networks Prof. B
4 Networks Prof. B

Functional Dependencies:

  1. (student_id, course_id) → instructor ✅ (composite primary key)
  2. course_id → instructor ❌ (Problem)
  • Here, course_id is not a superkey.
  • This violates BCNF, though the table is still in 3NF.

3. 🚀 Converting to BCNF

To remove this violation, we split the table:

Table 1: courses_instructors

course_id instructor
DBMS Prof. A
Networks Prof. B

Table 2: student_courses_bcnf

student_id course_id
1 DBMS
2 DBMS
3 Networks
4 Networks

4. ✅ Verification of BCNF

  • courses_instructors: course_id is the key, and all attributes depend on it directly.
  • student_courses_bcnf: (student_id, course_id) is the key, and no other dependency exists.

Thus, both tables are in BCNF.

5. 📊 Summary Table

Normal Form Rule
1NF Atomic values (no repeating groups)
2NF No partial dependency on part of a composite key
3NF No transitive dependency (non-key → non-key)
BCNF For every FD X → Y, X must be a superkey
