📘 DBMS NOTES (Well‑Structured & Exam Ready)

📘 DBMS (Database Management System)

A DBMS is software that is used to store, manage, and retrieve data from a database in an organized way.
It helps users create, update, delete, and access data easily and securely.

👉 Example: MySQL, Oracle, SQL Server

🐬 MySQL

MySQL is an open-source relational database management system (RDBMS).
It stores data in the form of tables (rows and columns) and uses SQL to manage data.

👉 Commonly used in web and application development.

💬 SQL (Structured Query Language)

SQL is a standard language used to communicate with a database.
It is used to create tables, insert data, update data, delete data, and retrieve data from databases.

👉 Example commands: SELECT, INSERT, UPDATE, DELETE

🎯 One-line Interview Summary

DBMS: Software to manage databases

MySQL: A relational database system

SQL: Language used to interact with databases

📂 SQL Command Types (Interview Notes)

SQL commands are divided into five main categories.

🏗️ DDL (Data Definition Language)

Used to define or change the structure of database objects.

Commands:

  • CREATE → Creates database or table
  • ALTER → Modifies table structure
  • DROP → Deletes table or database
  • TRUNCATE → Removes all records (structure remains)
  • RENAME → Renames a table

Example:

CREATE TABLE Student(id INT, name VARCHAR(50));

📝 DML (Data Manipulation Language)

Used to manipulate data inside tables.

Commands:

  • INSERT → Adds new data
  • UPDATE → Modifies existing data
  • DELETE → Removes data

Example:

INSERT INTO Student VALUES (1, 'Rahul');

🔍 DQL (Data Query Language)

Used to retrieve data from the database.

Command:

  • SELECT

Example:

SELECT * FROM Student;

🔐 DCL (Data Control Language)

Used to control access and permissions.

Commands:

  • GRANT → Gives permission
  • REVOKE → Removes permission

Example:

GRANT SELECT ON Student TO user1;

🔄 TCL (Transaction Control Language)

Used to manage transactions in the database.

Commands:

  • COMMIT → Saves changes permanently
  • ROLLBACK → Undoes changes
  • SAVEPOINT → Creates a rollback point

Example:

ROLLBACK;

🎯 One-Line Interview Trick

  • DDL → Structure
  • DML → Data
  • DQL → Fetch data
  • DCL → Permission
  • TCL → Transactions









🔹 TOPIC 1: DEPENDENCY

1. Functional Dependency (FD)

A functional dependency X → Y means attribute X uniquely determines attribute Y.

Example

RollNo → Name

2. Types of Functional Dependency

1. Trivial Dependency

A → A

2. Non‑Trivial Dependency

A → B

3. Full Functional Dependency

  • 📌 Attribute depends on whole composite key
(A, B) → C

4. Partial Dependency

  • ⚠️ Depends on part of composite key
(A, B) → C
A → C

5. Transitive Dependency

A → B
B → C
A → C

3. Candidate Key

A candidate key is a minimal set of attributes that uniquely identify a tuple.

Types of Keys

  • 🔑 Super Key
  • 🗝️ Candidate Key
  • ⭐ Primary Key
  • 🔁 Alternate Key

🔐 Super Key

  • A Super Key is any key that can uniquely identify a record in a table.
    It may contain extra attributes.

Example:
StudentID
StudentID + Name

🗝️ Candidate Key

  • A Candidate Key is the smallest possible super key.
  • It uniquely identifies a record without extra attributes.

Example:
StudentID, Email

⭐ Primary Key

  • A Primary Key is the candidate key chosen to identify records in a table.
    It cannot be NULL or duplicate.

Example:
StudentID

4. Multivalued Dependency (MVD)

A multivalued dependency occurs when one attribute determines multiple independent attributes.

Notation

A ↠ B

Example

  • 📞 Employee has multiple phone numbers
  • 🧠 Employee has multiple skills

🔹 TOPIC 2: NORMALIZATION

Anomalies in DBMS (Interview Notes)

Anomalies in DBMS are problems that occur when a database table is poorly designed and contains redundant data. These problems arise during normal database operations.

Types of Anomalies

  1. Insertion Anomaly ➕
    Occurs when we cannot insert data without inserting unnecessary or unrelated data.

  2. Update Anomaly 🔁
    Occurs when the same data is present in multiple rows, and updating it in one place but not everywhere leads to inconsistency.

  3. Deletion Anomaly ❌
    Occurs when deleting a record unintentionally removes important information.

Cause

  • ⚠️ Poor table design
  • 📄 Data redundancy
  • 🚫 Lack of normalization

Solution

  • ✅ Apply database normalization
  • 🗂️ Divide data into multiple related tables
  • 🔑 Use primary and foreign keys

One-Line Interview Answer

Anomalies are errors caused by data redundancy during insert, update, or delete operations in DBMS.

1. Normalization

Normalization is the process of organizing data to:

  • ♻️ Remove redundancy
  • ⚠️ Avoid anomalies
  • 🔐 Improve data integrity

2. First Normal Form (1NF)

Rules

A table is in First Normal Form (1NF) if:

  1. 🔹 Atomic values
  2. 🚫 No repeating groups
  3. 🎯 Single‑valued attributes

❌ Not 1NF

Phone = 123, 456

✅ 1NF

Phone = 123
Phone = 456

3. Second Normal Form (2NF)

Rules

A table is in Second Normal Form (2NF) if:

  1. ✅ Must be in 1NF
  2. ❌ No partial dependency

👉 Partial Dependency means:

A non-key attribute depends on part of a composite primary key

4. Third Normal Form (3NF)

Rules

A table is in Third Normal Form (3NF) if:

  1. ✅ Must be in 2NF
  2. ❌ No transitive dependency

👉 Transitive Dependency:

A non-key attribute depends on another non-key attribute

5. Boyce Codd Normal Form (BCNF)

Rule

A table is in BCNF if:

  • It is already in 3NF, and

  • For every functional dependency X → Y, X must be a super key

For every dependency X → Y, X must be a super key.
“X functionally determines Y”

6. Fourth Normal Form (4NF)

A table is in Fourth Normal Form (4NF) if:
Rules

  • ✅ Must be in BCNF
  • ❌ No multivalued dependency

7. Lossless Join Decomposition

After decomposition, if original table can be perfectly reconstructed, it is lossless.

8. Dependency Preservation

All functional dependencies should be preserved after decomposition without joining tables.

🔁 Normalization Flow 

1NF → 2NF → 3NF → BCNF → 4NF

Transactions

  • A sequence of operations performed as a single logical unit of work.

ACID Properties in DBMS

ACID properties ensure that database transactions are reliable, safe, and consistent, especially when multiple users access the database at the same time.

1. Atomicity

Meaning: All or nothing.

  • 🔹 A transaction is treated as a single unit.
  • 🔹 Either all operations succeed or none are applied.
  • 🔹 If any part fails, the entire transaction is rolled back.

Example:

  • 💸 Transfer ₹5,000 from Account A to Account B
  • ✅ Debit from A
  • ✅ Credit to B
  • ❌ If credit fails → debit is rolled back

2. Consistency

Meaning: Data must remain valid.

  • 📌 A transaction moves the database from one valid state to another.
  • 📌 All constraints, rules, triggers, and data types must be followed.

Examples:

  • 🚫 Balance cannot be negative
  • 🔑 Primary key must be unique

3. Isolation

Meaning: Transactions work independently.

  • 🔁 Multiple transactions can run at the same time.
  • 🧍 Each transaction behaves as if it is the only one running.
  • 🛑 Prevents problems like dirty reads and incorrect data.

Example:

  • 👥 Two users updating the same account should not see each other’s incomplete data.

4. Durability

Meaning: Data is permanent after commit.

  • 💾 Once a transaction is committed, it is permanently saved.
  • 🔒 Data remains safe even after system crash or power failure.

Example:

  • 🔄 After a successful money transfer, the updated balance remains even if the system restarts.

Interview One-Liner

ACID properties ensure reliable database transactions by guaranteeing Atomicity, Consistency, Isolation, and Durability.

Easy Way to Remember

  • A → 🔐 All or nothing
  • C → ✔️ Correct data
  • I → 🧍 Independent transactions
  • D → 💾 Data is permanent

🔹 TOPIC 3: ER DIAGRAM (Entity Relationship Model)

1. ER Model

The Entity Relationship (ER) Model is a conceptual model used to design a database by representing real‑world objects and their relationships.

Main Components

  • 🧩 Entity
  • 🏷️ Attribute
  • 🔗 Relationship

Advantages

  • ✅ Simple and easy to understand
  • 🌍 Represents real‑world data clearly
  • 🛠️ Useful for database design
  • 🔄 Easy to convert into relational model

2. Entity

An entity is a real‑world object that can be uniquely identified.

Types of Entity

1. Strong Entity

  • 🔑 Has its own primary key
  • 🧱 Exists independently
  • 📌 Example: Student, Book

2. Weak Entity

  • ❌ Does not have its own primary key
  • 🔗 Depends on a strong entity
  • 📌 Example: Edition (depends on Book)

3. Attributes

Attributes describe the properties of an entity.

Types of Attributes

  1. 🗝️ Key (Identifying) Attribute – Uniquely identifies an entity (Roll No, Book_ID)
  2. 📝 Descriptive Attribute – Does not uniquely identify (Name, City)
  3. 🧩 Simple Attribute – Cannot be divided further (Age)
  4. 🧱 Composite Attribute – Can be divided (Address → City, State, Pin)
  5. 🎯 Single‑Valued Attribute – Only one value (DOB)
  6. 📚 Multi‑Valued Attribute – Multiple values (Phone, Email)
  7. 💾 Stored Attribute – Stored in DB (DOB)
  8. 🧮 Derived Attribute – Calculated (Age from DOB)

4. Relationship

A relationship shows how entities are connected.

Types of Relationships

  1. 🔁 Unary – Entity related to itself
  2. 🔗 Binary – Relationship between two entities
  3. 🔀 Ternary – Relationship among three entities

5. Cardinality

Defines how many entities can participate in a relationship.

  • 1️⃣↔️1️⃣ One to One (1:1)
  • 1️⃣➡️♾️ One to Many (1:M)
  • ♾️↔️♾️ Many to Many (M:N)

6. ER Diagram Example: Online Book Database

Step 1: Entities

  • 👤 Author
  • 📘 Book
  • 🏢 Publisher

Step 2: Relationships & Cardinality

  • 🧐 Author reviews Book (M:N)
