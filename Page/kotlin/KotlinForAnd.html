🟤 KOTLIN CONCURRENCY

🔰 Benefits of Kotlin Concurrency

Kotlin concurrency tools (Coroutines & Flow) help you write asynchronous, non-blocking, and scalable code in a simple and readable way.

✅ Key Benefits

  • Lightweight compared to threads
  • Improves app responsiveness (especially Android)
  • Easier async code with sequential style
  • Built-in cancellation support
  • Structured and safer concurrency

2️⃣3️⃣ Coroutines (VERY IMPORTANT)

🔹 What are Coroutines

Coroutines are lightweight threads that allow you to run long-running tasks (network, database, file I/O) without blocking the main thread.

They run asynchronously but look like normal sequential code.

🔹 suspend Function

A suspend function can pause and resume execution without blocking a thread.

suspend fun fetchData(): String {
    delay(1000)
    return "Data loaded"
}

👉 Can only be called from another suspend function or coroutine.

🔹 Coroutine Builders

▫️ launch

  • Starts a coroutine
  • Does not return a result
launch {
    fetchData()
}

▫️ async

  • Starts a coroutine
  • Returns a Deferred<T> (future result)
val result = async { fetchData() }
println(result.await())

🔹 Dispatchers

Dispatchers decide which thread the coroutine runs on.

  • Dispatchers.Main → UI thread
  • Dispatchers.IO → Network, DB, file operations
  • Dispatchers.Default → CPU-intensive work
launch(Dispatchers.IO) {
    fetchData()
}

🔹 Job & Deferred

  • Job → Controls coroutine lifecycle (cancel, join)
  • Deferred → Job with result
val job: Job = launch { }
job.cancel()

🔹 Structured Concurrency

Ensures child coroutines are cancelled automatically when the parent is cancelled.

coroutineScope {
    launch { }
    launch { }
}

🔹 Coroutine Scopes

Defines the lifecycle of coroutines.

  • GlobalScope ❌ (avoid)
  • viewModelScope (Android)
  • lifecycleScope (Android)
viewModelScope.launch {
    fetchData()
}

🔹 Exception Handling

Use try-catch or CoroutineExceptionHandler.

val handler = CoroutineExceptionHandler { _, e ->
    println(e.message)
}

launch(handler) { }

2️⃣4️⃣ Flow

Flow is used to handle streams of data asynchronously.

🔹 Cold Flow

  • Starts only when collected
  • Emits values fresh for each collector
val flow = flow {
    emit(1)
    emit(2)
}

🔹 Hot Flow

  • Emits values continuously
  • Shared among collectors

Examples: StateFlow, SharedFlow

🔹 flow

Builder to create a flow.

val dataFlow = flow {
    emit("A")
    emit("B")
}

🔹 collect

Collects values emitted by a flow.

dataFlow.collect {
    println(it)
}

🔹 Operators (map, filter) 

flow
    .map { it * 2 }
    .filter { it > 2 }
    .collect { println(it) }

🔹 StateFlow & SharedFlow

▫️ StateFlow

  • Holds the latest value
  • Used for UI state
val state = MutableStateFlow(0)

▫️ SharedFlow

  • Emits events
  • No default value
val events = MutableSharedFlow<String>()

✅ Summary

  • Coroutines handle async tasks efficiently
  • Flow manages continuous data streams
  • StateFlow is best for UI state
  • SharedFlow is best for one-time events






⚫ KOTLIN ANDROID / JVM SPECIFIC

🔰 Introduction

These topics explain how Kotlin works under the hood on the JVM and how it interacts with Java. They are important for Android development, performance understanding, and senior-level interviews.

2️⃣5️⃣ Annotations & Reflection

🔹 Built-in Annotations

Kotlin provides many built-in annotations used by the compiler and runtime.

Common examples:

  • @Deprecated → Marks code as deprecated
  • @Suppress → Suppresses compiler warnings
  • @JvmStatic → Creates static methods for Java
  • @JvmField → Exposes field directly to Java
@Deprecated("Use newMethod()")
fun oldMethod() {}

🔹 Custom Annotations

You can create your own annotations.

@Target(AnnotationTarget.CLASS, AnnotationTarget.FUNCTION)
@Retention(AnnotationRetention.RUNTIME)
annotation class LogExecution

👉 Commonly used in frameworks and libraries.

🔹 Kotlin Reflection Basics

Reflection allows inspection of classes, functions, and properties at runtime.

val kClass = String::class
println(kClass.simpleName)

⚠️ Reflection is powerful but slower; use carefully.

2️⃣6️⃣ Kotlin JVM Internals

🔹 Bytecode Generation

Kotlin code is compiled into JVM bytecode, just like Java.

  • Kotlin → Bytecode → JVM
  • Fully compatible with Java

🔹 Kotlin vs Java Bytecode

Aspect Kotlin Java
Null safety Yes No
Default arguments Yes No
Data classes Yes No

👉 Kotlin generates extra bytecode for safety features.

🔹 SAM Conversions

SAM = Single Abstract Method (Java interfaces).

Kotlin allows passing lambdas instead of anonymous classes.

button.setOnClickListener {
    println("Clicked")
}

2️⃣7️⃣ Interoperability with Java

Kotlin is designed for 100% Java interoperability.

🔹 Calling Java from Kotlin 

val list = ArrayList<String>()
list.add("Hello")

Kotlin treats Java code naturally.

🔹 Calling Kotlin from Java

Kotlin code may look different in Java due to default parameters and properties.

🔹 @JvmStatic

Makes functions behave like static methods in Java.

class Utils {
    companion object {
        @JvmStatic
        fun greet() {}
    }
}

🔹 @JvmOverloads

Generates overloaded methods for Java.

fun greet(name: String = "User") {}

🔹 @JvmField

Exposes Kotlin property as a field for Java.

@JvmField
val VERSION = "1.0"

✅ Summary

  • Annotations add metadata to code
  • Reflection inspects code at runtime
  • Kotlin compiles to JVM bytecode
  • Interoperability with Java is seamless





🧠 KOTLIN DESIGN & BEST PRACTICES

🔰 Introduction

Design patterns and best practices help you write clean, maintainable, scalable, and testable Kotlin code. These topics are very important for Android architecture discussions and interviews.

2️⃣8️⃣ Design Patterns in Kotlin

🔹 Singleton Pattern

Ensures that only one instance of a class exists.

Kotlin provides a built-in way using object.

object UserManager {
    fun login() {
        println("User logged in")
    }
}

👉 Thread-safe by default.

🔹 Factory Pattern

Used to create objects without exposing the creation logic.

sealed class Shape {
    class Circle : Shape()
    class Square : Shape()
}

object ShapeFactory {
    fun create(type: String): Shape {
        return when (type) {
            "circle" -> Shape.Circle()
            else -> Shape.Square()
        }
    }
}

🔹 Builder Pattern

Used to create complex objects step by step.

class User private constructor(
    val name: String,
    val age: Int
) {
    class Builder {
        private var name: String = ""
        private var age: Int = 0

        fun setName(name: String) = apply { this.name = name }
        fun setAge(age: Int) = apply { this.age = age }

        fun build() = User(name, age)
    }
}

🔹 Observer Pattern

Used when multiple objects need to react to changes.

interface Observer {
    fun update(data: String)
}

class Subject {
    private val observers = mutableListOf<Observer>()

    fun register(observer: Observer) {
        observers.add(observer)
    }

    fun notifyObservers(data: String) {
        observers.forEach { it.update(data) }
    }
}

👉 In Android, Flow and LiveData follow this pattern.

🔹 Repository Pattern

Separates data layer from business logic.

class UserRepository {
    fun getUser(): String {
        return "User Data"
    }
}

👉 Commonly used with ViewModel in MVVM.

2️⃣9️⃣ Kotlin Best Practices

🔹 Immutability

Prefer immutable (val) over mutable (var).

val name = "Nafis"  // Preferred

Benefits:

  • Safer code
  • Easier debugging

🔹 Functional Programming Style

Use higher-order functions instead of loops.

val even = list.filter { it % 2 == 0 }

Benefits:

  • Cleaner code
  • Fewer bugs

🔹 Avoiding !!

!! forces null and may crash the app.

// Avoid this
val len = name!!.length

// Prefer this
val len = name?.length ?: 0

🔹 Performance Considerations

  • Avoid unnecessary object creation
  • Use inline for heavy lambdas
  • Prefer Sequence for large collections
  • Avoid reflection in performance-critical code

✅ Summary

  • Design patterns improve structure and reusability
  • Kotlin reduces boilerplate for patterns
  • Best practices lead to safer and faster apps
