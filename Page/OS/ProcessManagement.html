🧠 Process Management – Complete Notes (Operating Systems)

1. What is a Process?

A process is a program in execution along with its current state, resources, and execution context.

📌 One-line (GATE / Interview):

Program = passive, Process = active

2. Program vs Process (Very Important)

Program Process
Static (stored on disk) Dynamic (in execution)
No state Has state
Single entity Multiple processes possible
No resources Owns resources

3. Process States

3.1 Five-State Process Model

  1. 🆕 New – Process is being created
  2. ⏳ Ready – Waiting for CPU allocation
  3. ▶️ Running – Executing on CPU
  4. ⏸️ Waiting / Blocked – Waiting for I/O or event
  5. ❌ Terminated – Execution finished

📌 GATE Note: Scheduling occurs between Ready ↔ Running

4. Process Control Block (PCB) ⭐⭐⭐

PCB is a data structure maintained by OS to store all information about a process.

4.1 Information stored in PCB

  • 🆔 Process ID (PID)
  • 📌 Process State
  • 🧮 Program Counter
  • 🧱 CPU Registers
  • ⏱️ CPU Scheduling Information
  • 📦 Memory Management Information
  • 📂 I/O Status Information
  • 🧾 Accounting Information

📌 GATE Trap: PCB is stored in main memory, NOT on disk

5. Context Switching

Context switching is the process of saving the state of one process and loading the state of another process.

  • Causes overhead
  • No useful work during switching

📌 Dispatcher performs context switching

6. Process Scheduling Queues

  • 📥 Job Queue – All processes in the system
  • ⏳ Ready Queue – Processes ready to execute
  • ⏸️ Device Queue – Waiting for I/O devices

7. Types of Processes

7.1 Based on Execution

  • 👤 User Process
  • ⚙️ System Process

7.2 Based on Interaction

  • 🧑‍💻 Interactive Process
  • 📦 Batch Process

8. Process Creation

Processes are created using system calls.

8.1 UNIX Process Creation

  • fork() → creates child process
  • exec() → replaces process image
  • wait() → parent waits for child
  • exit() → terminates process

📌 Interview Favorite: fork() returns 0 to child and PID to parent

9. Process Termination

A process terminates when:

  • It finishes execution normally
  • Parent terminates child
  • Fatal error occurs

10. Orphan and Zombie Processes ⭐⭐

10.1 Orphan Process

  • Parent terminates before child
  • Child adopted by init process

10.2 Zombie Process

  • Child terminates but parent doesn’t collect exit status
  • Entry remains in process table

📌 Interview Trap: Zombie consumes PCB only, not CPU

11. Inter-Process Communication (IPC)

Mechanism to allow processes to communicate and synchronize.

11.1 IPC Methods

  • 📬 Pipes
  • 📪 Message Queues
  • 🧠 Shared Memory
  • 🔐 Semaphores
  • 🧷 Signals

📌 GATE Note: Shared memory is fastest IPC

12. Threads vs Processes ⭐⭐⭐

Process Thread
Heavyweight Lightweight
Separate address space Shared address space
High context switch cost Low context switch cost
Independent Dependent

📌 Interview Line: Threads share code, data, and files

13. Multithreading Models

  • 1️⃣ One-to-One
  • 2️⃣ Many-to-One
  • 3️⃣ Many-to-Many

📌 Modern OS uses: One-to-One

14. Critical Section Problem ⭐⭐⭐

Critical section is a part of code where shared resources are accessed.

Requirements:

  1. Mutual Exclusion
  2. Progress
  3. Bounded Waiting

15. Deadlock (Relation to Process Management)

Deadlock occurs when processes are waiting indefinitely for resources.

Coffman Conditions:

  • Mutual Exclusion
  • Hold and Wait
  • No Preemption
  • Circular Wait

16. GATE Important One-Liners

  • PCB is created when process is created
  • Context switching is pure overhead
  • Ready queue may be implemented using linked list
  • fork() creates identical copy initially
  • Zombie process is not dangerous

17. Interview-Oriented Questions

  • Difference between process and thread?
  • What is PCB?
  • Why context switching is costly?
  • What happens if parent dies before child?
  • Why shared memory is fastest IPC?

18. Summary (Last-Day Revision)

  • Process = program in execution
  • PCB stores everything about process
  • fork() creates, exec() replaces
  • Zombie ≠ Orphan
  • Threads are lightweight

☠️ Deadlocks – Complete Notes (GATE & Interview Oriented)

1. What is a Deadlock?

A deadlock is a situation in which a set of processes are blocked forever because each process is holding a resource and waiting for another resource held by some other process.

📌 One‑line (GATE / Interview):

Deadlock occurs when processes wait indefinitely for resources held by each other.

2. Necessary Conditions for Deadlock ⭐⭐⭐ (Coffman Conditions)

Deadlock can occur if and only if all four conditions hold simultaneously.

  1. 🔒 Mutual Exclusion
    At least one resource must be non‑sharable.

  2. 🤝 Hold and Wait
    A process holds at least one resource and waits for additional resources.

  3. 🚫 No Preemption
    Resources cannot be forcibly taken away from a process.

  4. 🔁 Circular Wait
    A circular chain of processes exists, where each process waits for a resource held by the next.

📌 GATE Trick:
If any one condition is removed, deadlock cannot occur.

3. Resource Allocation Graph (RAG)

A Resource Allocation Graph is a directed graph used to represent allocation and request of resources.

3.1 Components

  • 🟢 Process nodes → Circle (P1, P2 …)
  • 🔵 Resource nodes → Rectangle (R1, R2 …)
  • ➡️ Request edge (P → R)
  • ⬅️ Assignment edge (R → P)

3.2 Deadlock Detection using RAG

  • 🔴 If graph has a cycle and each resource has single instance → Deadlock exists
  • 🟡 Cycle present + multiple instances → Deadlock may or may not exist

📌 GATE Favorite Statement:

Cycle in RAG is a necessary but not sufficient condition for deadlock (multiple instances case)

4. Safe State vs Unsafe State ⭐⭐⭐

4.1 Safe State

A system is in a safe state if there exists at least one sequence of processes such that each process can complete execution without leading to deadlock.

➡️ This sequence is called Safe Sequence.

4.2 Unsafe State

A system is in an unsafe state if no safe sequence exists.

📌 Important Point (GATE Trap):

  • Unsafe state does NOT always mean deadlock
  • But deadlock always implies unsafe state

5. Handling Deadlocks (Strategies)

  1. ❌ Deadlock Prevention
    Ensure at least one Coffman condition never holds.

  2. 🚨 Deadlock Avoidance
    Decide dynamically whether allocation is safe (Banker’s Algorithm).

  3. 🔍 Deadlock Detection & Recovery
    Detect deadlock and recover by killing processes or preempting resources.

  4. 😴 Ignore Deadlock
    Used in general‑purpose OS ("Ostrich Algorithm").

6. Banker’s Algorithm ⭐⭐⭐

Used for deadlock avoidance. System checks whether granting a request keeps the system in a safe state.

6.1 Data Structures

  • 📊 Available – Free resources
  • 📋 Max – Maximum demand
  • 📦 Allocation – Currently allocated resources
  • 📐 Need = Max − Allocation

6.2 Banker’s Algorithm – Steps

  1. Check if request ≤ Need
  2. Check if request ≤ Available
  3. Pretend to allocate resources
  4. Run safety algorithm to check safe state
  5. If safe → allocate, else → deny

6.3 Light Numerical (GATE Level)

If after allocation a safe sequence exists, then:

✅ System is safe → allocation allowed
❌ No safe sequence → allocation denied

📌 Exam Tip:
Never fully calculate if clearly unsafe in first iterations.

7. Deadlock vs Starvation ⭐⭐

Deadlock Starvation
Circular wait Resource never allocated
Permanent Can be resolved by scheduling
Involves multiple processes Can involve single process

8. GATE One‑Liners (Must Revise)

  • All four Coffman conditions are necessary for deadlock
  • Cycle + single instance → deadlock guaranteed
  • Banker’s algorithm avoids deadlock
  • Unsafe ≠ Deadlock
  • Deadlock is permanent until external action
