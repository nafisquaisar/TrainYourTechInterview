1️⃣ Project Overview & Basic Questions

Q1. Can you explain your TuneLyf project in simple words?

Answer:
TuneLyf is an Android music application that supports both online streaming and offline music stored on the user’s device.
The idea came from a real problem I faced. Most music apps focus either on online streaming or on offline songs, so users often need more than one app.

To solve this, I built a single app where users can stream music online, play songs that are stored on their device, create playlists, and listen to music smoothly in the background.
The app uses Firebase for secure login and follows the MVVM architecture to ensure stable playback and a good user experience.

Q2. What problem does TuneLyf solve?

3️⃣ MVVM Architecture (Very Important for TCS)

Q16. What is MVVM architecture?

Answer:
MVVM is an Android architecture pattern that separates the UI from business logic.
It divides the app into Model, View, and ViewModel, which makes the code clean, modular, and easy to test.

Q17. Why did you use MVVM in TuneLyf?

Answer:
It keeps the code clean, handles configuration changes, and improves maintainability.

Q18. What is the role of Model, View, and ViewModel?

Answer:

  • 📦 Model: Handles data and business logic
  • 🖥️ View: Displays UI
  • 🧠 ViewModel: Holds and manages UI data

Q19. How does View communicate with ViewModel?

Answer:
The View communicates with the ViewModel by calling functions in the ViewModel, and the ViewModel sends data back to the View using LiveData or StateFlow, which the View observes.

The View calls ViewModel methods, and observes data from ViewModel using LiveData or StateFlow.

Q20. How does ViewModel survive configuration changes?

Answer:
ViewModel survives configuration changes because it is not destroyed when the screen is recreated, such as during screen rotation.
Android keeps the same ViewModel instance and attaches it to the new View, so the UI data remains intact.

Example (TuneLyf):

When the screen rotates while a song is playing, the MusicViewModel still holds the current song and playback state, so the music and UI continue without interruption.

Q21. What is LiveData?

Answer:
LiveData is an observable data holder that is lifecycle-aware.
It means the UI automatically gets updates only when it is active, which helps prevent crashes and memory leaks.

Example (TuneLyf):

LiveData is used to observe the currently playing song or playback state, so when the data changes in the ViewModel, the UI updates automatically.

Q22. Difference between LiveData and MutableLiveData?

Answer:
LiveData is read-only, MutableLiveData allows updates.

Q23. How did MVVM help in scalability?

Answer:
It allowed adding new features without breaking existing code.

Q24. What problems occur if we don’t use MVVM?

Answer:
Tight coupling, difficult testing, and poor maintainability.

Q25. Have you used any single source of truth concept?

Answer:
Yes, I used ViewModel as the single source of truth to keep playback state consistent across the app.

4️⃣ Retrofit & API Integration

Q26. What is Retrofit?

Answer:

  • Retrofit is a type-safe HTTP client for Android that simplifies making API calls.
    It lets us define API endpoints using annotations and automatically converts JSON responses into Kotlin objects.

  • Retrofit uses OkHttp internally to handle the actual network communication such as sending requests, receiving responses, and managing timeouts.

  • Retrofit is used to define and call APIs easily, while OkHttp handles the actual network operations behind the scenes.

Q27. Why did you use Retrofit instead of Volley?

Answer:
Retrofit is easier to manage, type-safe, and integrates well with Kotlin and MVVM.

Q28. How does Retrofit work internally?

Answer:
Retrofit converts API methods into HTTP requests, uses OkHttp to execute them, and converts the response into Kotlin objects.

Q29. What is an API?

Answer:
An API is a set of rules that allows a client application to communicate with a server.
It defines how a request is sent and how the response is received.

Example:

In TuneLyf, the app requests songs from a server through an API, and the server sends the music data back.

Q30. What is JSON?

Answer:
JSON (JavaScript Object Notation) is a lightweight data format used to exchange data between a client and a server.
It is easy for humans to read and easy for machines to parse.

Q31. How did you parse JSON response?

Answer:
I parsed the JSON response using Retrofit with a converter like Gson.
I created Kotlin data classes that match the JSON structure, and Retrofit automatically converted the JSON response into Kotlin objects.

Example (TuneLyf):

When the API returns a list of songs in JSON format, Retrofit maps it directly to a list of Song data class objects.

Q32. What are @GET, @POST, @Query, @Path?

Answer:
These are Retrofit annotations used to define API requests.

@GET
  • Used to fetch data from the server.

Example: Get a list of songs.

@POST
  • Used to send data to the server.

Example: Login or send user details.

@Query
  • Used to pass data in the URL as query parameters.

Example: ?artist=Arijit

@Path
  • Used to pass dynamic values directly in the URL path.

Example: /songs/{id}

6️⃣ Offline Playback & Room Database

SQLite vs Room (Easy Explanation)

SQLite

  • SQLite is a local database engine
  • We write raw SQL queries (CREATE, INSERT, SELECT)
  • More boilerplate code
  • No compile-time query checking
  • Higher chance of runtime errors

Example:
You manually write SQL queries to store and fetch data.

Room

  • Room is a library built on top of SQLite by Google
  • Uses annotations instead of raw SQL
  • Less code and clean structure
  • Compile-time checking of SQL queries
  • Easy integration with LiveData / Flow
  • Room internally uses SQLite, but makes it safer and easier.

Q46. What is Room Database?

Answer:
Room is an abstraction layer over SQLite.

Q47. Why Room instead of SQLite?

Answer:
Room is safer, simpler, and supports LiveData.
SQLite is a lightweight local database that stores data on the device without needing a server.

Aspect Service Foreground Service

User awareness User may not know it is running User always knows it is running

Spotify vs TuneLyf – Interview Comparison

🔹 1. Core Purpose

  • Spotify: Streaming-first music platform focused on online content and recommendations.
  • TuneLyf: Hybrid music application supporting both online streaming and device music.

🔹 2. Device (Local) Music Support

  • Spotify: Limited and unreliable support for local files, mainly through desktop sync.
  • TuneLyf: Full native support for device music using Android MediaStore.

🔹 3. Offline Usage

  • Spotify: Offline playback only for downloaded streaming songs.
  • TuneLyf: Works fully offline for device music without any download requirement.

🔹 4. Android Storage Handling

  • Spotify: Avoids deep storage access due to Android scoped storage restrictions.
  • TuneLyf: Properly handles scoped storage and permissions to access local audio files.

🔹 5. Playback System

  • Spotify: Uses a proprietary player with DRM and backend-controlled playback.
  • TuneLyf: Uses ExoPlayer with a foreground service for stable and continuous playback.

🔹 6. Architecture Pattern

  • Spotify: Closed and backend-driven architecture.
  • TuneLyf: Clean MVVM architecture with a single shared ViewModel for playback state.

🔹 7. Internet Dependency

  • Spotify: Requires internet for most features except cached downloads.
  • TuneLyf: Can operate without internet when playing device music.

🔹 8. Business Focus

  • Spotify: Monetizes through subscriptions and advertisements, prioritizing streaming.
  • TuneLyf: User-focused design prioritizing flexibility and real-world usage.

✅ One-Line Interview Answer

Spotify is a streaming-first platform with limited local file support, while TuneLyf is a hybrid Android music app that supports both online streaming and native device music with full offline usability.
